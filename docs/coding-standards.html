总则

所有语言通用的核心原则

🎯 可读性 > 简洁性

代码被阅读的次数远多于编写的次数。宁可多写几行清晰的代码，不要写一行没人看懂的"聪明"代码。

🧩 单一职责

每个文件、函数、模块只做一件事。如果你需要用"和"来描述一个函数的功能，它就该拆分了。

📏 一致性 > 个人偏好

团队内风格统一比"哪种更好"更重要。所有争议以本规范为准，规范未覆盖的以现有代码风格为准。

🔍 显式 > 隐式

不要靠"魔法"——状态变更、副作用、隐式类型转换都是隐患。让代码意图一目了然。

规范等级说明

  • 强制 必须遵守，违规即 Bug
  • 推荐 应当遵守，特殊情况可偏离但需注释说明
  • 建议 优秀实践，按场景采纳

禁止缩写——usr → user，btn → button，cfg → config

5.6 文件组织原则 强制

ST-01

按功能分组优于按类型分组——feature/market/components/ 优于 components/market/

ST-02

就近原则——组件、样式、测试、类型尽量放在同一个目录

ST-03

index 作为导出面——每个模块通过 index.ts/mod.rs/__init__.py 统一导出

ST-04

禁止循环依赖——模块 A 依赖 B，B 不能再依赖 A

ST-05

公共代码下沉——被 3 个以上模块共用的代码提取到公共层

依赖管理

引入什么、怎么引入、版本锁定

6.1 引入依赖的原则 强制

D-01

能用标准库就不用第三方——标准库稳定、无供应链风险

D-02

一个功能只引入一个库——日期只用 dayjs，不要同时用 moment + dayjs

D-03

新增依赖必须评估——检查：周下载量、最后更新时间、issue 数、license、bundle size

D-04

锁文件必须提交——package-lock.json / go.sum / Cargo.lock / poetry.lock

D-05

禁止依赖未维护的包——6个月无更新 + 大量未处理 issue = 不用

6.2 依赖评估清单 推荐

检查项通过标准
周下载量> 10,000
最后更新< 6 个月
Open Issues< 50（相对 Star 数）
LicenseMIT / Apache-2.0 / BSD
Bundle Size (前端)< 50KB gzipped
类型支持 (TS)内置类型声明 或 @types/*
安全漏洞npm audit / go vulncheck 无高危

错误处理

不要吞错误、不要裸 panic、不要 ignore

7.1 通用规则 强制

E-01

禁止吞错误——catch(e) {} / except: pass / _ = mightFail() 绝对禁止

E-02

错误必须包含上下文——不要只抛 "operation failed"，要包含 "what operation, on what, why"

E-03

在能处理错误的层级处理——不要在底层吞掉，不要在上层无脑 re-throw

E-04

区分业务错误和系统错误——业务错误（参数校验）用 4xx，系统错误（数据库挂了）用 5xx

E-05

不要用错误做流程控制——"找不到"不是错误，是正常的业务分支

TypeScript TS

try {
  const data = await fetchData();
} catch (error) {
  logger.error('Failed to fetch user data', { userId, error });
  throw new AppError('FETCH_FAILED', { cause: error });
}
try {
  const data = await fetchData();
} catch (e) {
  // TODO: handle error
}

Go Go

data, err := fetchData()
if err != nil {
  return fmt.Errorf("fetch user %s: %w", userId, err)
}
data, _ := fetchData()

注释与文档

好的代码自解释，注释只解释 WHY

8.1 注释原则 强制

C-01

注释解释 WHY，不解释 WHAT——代码已经说了做什么，注释只说为什么这样做

C-02

公共 API 必须有文档注释——所有 export 的函数/类/接口必须有 JSDoc / godoc / doc comment

C-03

TODO 必须带信息——// TODO(zhangsan): 具体描述 + #issue编号

C-04

注释掉的代码直接删——有 Git 历史，不需要注释保留旧代码

C-05

特殊处理必须有注释——Hack、Workaround、兼容逻辑必须说明原因

// 降级到 v1 API，因为 v2 在 iOS 15 上有 CORS 问题 (#1234)

// 设置 URL

测试规范

不写测试的代码 = 不负责任的代码

9.1 测试要求 强制

T-01

覆盖率底线——核心逻辑 ≥ 80%，工具函数 ≥ 90%，UI 组件 ≥ 60%

T-02

测试命名——test('should return 404 when user not found')，不用 test('user test 1')

T-03

AAA 模式——Arrange（准备）→ Act（执行）→ Assert（断言）

T-04

每个测试独立——不依赖其他测试的执行顺序或副作用

T-05

PR 必须包含测试——新增功能/修复 Bug 必须有对应测试

安全规范

安全不是可选项

10.1 安全红线 强制

SEC-01

禁止硬编码密钥——API Key、密码、Token 必须通过环境变量或密钥管理服务注入

SEC-02

禁止 eval()——以及所有动态代码执行，没有例外

SEC-03

SQL 必须参数化——禁止字符串拼接 SQL

SEC-04

用户输入必须校验和转义——前后端都要，防止 XSS/注入

SEC-05

依赖安全审计——CI 中必须运行 npm audit / go vulncheck / cargo audit

SEC-06

敏感数据不进日志——密码、Token、身份证号等不能出现在日志中

Git 规范

提交历史也是文档

11.1 分支策略 推荐

分支用途命名
main生产代码，永远可部署—
develop开发集成分支—
功能分支新功能开发feat/agent-market
修复分支Bug 修复fix/login-error
发布分支版本发布准备release/v1.0.0
热修复线上紧急修复hotfix/payment-crash

11.2 Commit 规范 强制

采用 Conventional Commits 格式：

格式
<type>(<scope>): <description>

[可选] 详细说明
[可选] Breaking change / Close #xxx
Type用途示例
feat新功能feat(market): add agent search
fixBug 修复fix(auth): resolve token expiry issue
docs文档变更docs: update API reference
style格式调整（不影响逻辑）style: fix indentation
refactor重构（不是新功能也不是修 Bug）refactor(engine): extract install logic
perf性能优化perf(list): virtualize agent list
test添加/修改测试test(market): add search test
chore构建/工具链变更chore: upgrade dependencies

11.3 Commit 规则 强制

G-01

一个 commit 做一件事——不要把 3 个功能混在一个 commit

G-02

commit 前跑 lint 和测试——CI 会检查，别浪费 CI 时间

G-03

禁止 commit 敏感文件——.env、密钥文件、大型二进制文件

G-04

描述用现在时态——"add feature" 而非 "added feature"

G-05

PR 合并用 squash——保持 main 历史干净

性能规范

先写对，再写快

12.1 性能原则 推荐

P-01

不要过早优化——先保证正确，有性能问题时再优化，用 profiler 定位瓶颈

P-02

O(n²) 以下必须优化——循环内查询数据库、循环内网络请求绝对禁止

P-03

列表必须分页/虚拟化——不要一次加载 10000 条数据

P-04

大计算放后端——前端不做重计算，导出/分析等放后端异步处理

P-05

图片/资源懒加载——不在首屏的图片、组件延迟加载

反模式黑名单

看到这些写法就改掉

反模式描述正确做法
🚫 God Object一个文件/类承担所有职责拆分为单一职责模块
🚫 Magic Number代码中出现 if (status === 3)用常量 if (status === Status.ACTIVE)
🚫 Callback Hell多层嵌套回调用 async/await 或 Promise 链
🚫 Copy-Paste Code相同逻辑复制多份提取公共函数
🚫 Premature Abstraction只有一个实现就抽接口有 3 个相似实现时再抽象
🚫 Global State全局可变状态随处修改状态集中管理，单向数据流
🚫 Leaky Abstraction抽象层暴露底层细节封装要完整，接口要干净
🚫 String Typing用字符串表示类型 type: 'user'用枚举/联合类型
🚫 Console.log 调试调试用的 log 提交到代码里用 logger，调试完删除
🚫 Any / interface{}逃逸类型检查定义明确的类型

快速自查清单

  • ✅ 文件行数是否超限？
  • ✅ 函数是否只做一件事？
  • ✅ 命名是否自解释？
  • ✅ 是否有吞错误的地方？
  • ✅ 是否有硬编码的密钥或 Magic Number？
  • ✅ 注释是否解释了 WHY？
  • ✅ 测试是否覆盖核心逻辑？
  • ✅ 依赖是否经过评估？
